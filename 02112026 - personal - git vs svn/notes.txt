Basic SVN Command Mapping
| Git                 | SVN          |
| ------------------- | ------------ |
| git clone           | svn checkout |
| git pull            | svn update   |
| git status          | svn status   |
| git add             | svn add      |
| git commit          | svn commit   |
| git log             | svn log      |
| git diff            | svn diff     |
| git checkout branch | svn switch   |
| git branch          | svn copy     |





Most SVN repos look like
project/
   trunk/
   branches/
   tags/






Git Workflow pt1 → SVN Equivalent
--- Git pt1 ---
git pull origin main
do changes
git add --all
git commit -m "..."
git push origin main
---------------
--- SVN Version ---
svn update
(make changes)
svn status
svn add <new files if any>
svn commit -m "..."
-------------------





Git Workflow pt2 (Feature Branching) → SVN Style
--- GIT ---
git checkout main
git pull origin main
git branch feature/new-feature
git checkout feature/new-feature
-----------
--- SVN ---
svn update
svn copy ^/trunk ^/branches/feature-new-feature -m "Create feature branch"
svn switch ^/branches/feature-new-feature
-----------
Work Normally
(make changes)
svn status
svn add <files>
svn commit -m "Work on feature"






Keep Branch Updated With Trunk
--- GIT ---
git checkout main
git pull
git checkout feature
git merge main
-----------
--- SVN ---
svn switch ^/trunk
svn update
svn switch ^/branches/feature-new-feature
svn merge ^/trunk
svn commit -m "Merge trunk into feature branch"
-----------
When Feature Is Done → Merge Back To Trunk
Equivalent of PR merge.
svn switch ^/trunk
svn update
svn merge ^/branches/feature-new-feature
svn commit -m "Merge feature-new-feature into trunk"







Your Git pt3 (Dev → Main → Prod)
- Most SVN shops do not use multiple long-lived branches like Git.
- Typical pattern:
	trunk = active development
	tags = releases
- Release Flow
	trunk → tag → deployed






Create Release Tag
--- GIT ---
git checkout production
git merge main
git push
-----------
--- SVN ---
svn copy ^/trunk ^/tags/release-2026-02-11 -m "Create release tag"
-----------






Practical Daily Workflow
--- SVN ---
svn update
(make fix)
svn status
svn commit -m "Fix printer log overflow"
-----------





Does SVN Have do Branching?
Yes.
	But culturally:
		Git: branch by default
		SVN: branch only when necessary
Git → cheap copies
SVN → expensive copies
So:
	Big feature → maybe branch
	Risky refactor → maybe branch
	Small bug fix → trunk
Most daily work = trunk.
